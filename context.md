Hound — AI-Native Test Automation Platform

Context
We're building Hound, an AI-powered E2E test automation platform inspired by Momentic.ai. The goal is to let developers and PMs author, run, and debug end-to-end tests using natural language and AI-powered element targeting — no CSS selectors or XPaths needed.

Key decisions: Web-first MVP, Next.js 15 App Router, Playwright for browser automation, Anthropic Claude for AI, PostgreSQL + Prisma, shadcn/ui + Tailwind.

```
## Implementation Phases

### Phase 1: Project Foundation
**Goal**: Running Next.js app with auth, database, and project management.

1. Initialize Next.js 15 with TypeScript, Tailwind, App Router
2. Set up Prisma with PostgreSQL schema
3. Add NextAuth.js with credentials provider
4. Build dashboard layout (sidebar, topbar) with shadcn/ui
5. Build project CRUD (create, list, view, settings)

**Files**: `package.json`, `prisma/schema.prisma`, `src/app/layout.tsx`, `src/lib/db.ts`, `src/lib/auth.ts`, all layout components, project pages and API routes.

### Phase 2: Test Authoring
**Goal**: Visual test editor where users can create tests with ordered steps.

1. Build test list page with create/edit/delete
2. Build test editor with:
   - Step palette (sidebar of available step types)
   - Drag-and-drop step ordering (using dnd-kit)
   - Step configuration forms per step type
   - Natural language input for AI steps
3. API routes for test and step CRUD

**Files**: `src/components/tests/test-editor/*`, test pages, test API routes.

### Phase 3: Execution Engine
**Goal**: Run tests using Playwright server-side and stream results in real-time.

1. Build `executor.ts` — the core orchestrator that:
   - Launches a Playwright browser
   - Iterates through test steps
   - Executes each step type via `step-handlers.ts`
   - Captures screenshots after each step
   - Records console/network logs
   - Writes StepResult records to DB
   - Emits SSE events for real-time UI updates
2. Build step handlers for each StepType (navigate, click, type, wait, assert, etc.)
3. Build SSE streaming endpoint (`/api/runs/[runId]/stream`)
4. Build run viewer UI with step timeline, screenshots, and logs

**Files**: `src/lib/engine/*`, run API routes, `src/components/runs/*`.

### Phase 4: AI Agent Layer
**Goal**: AI-powered element targeting, assertions, and failure analysis.

1. **Locator Agent** (`locator-agent.ts`):
   - Input: natural language description + page DOM (accessibility tree) + screenshot
   - Output: Playwright selector
   - Uses Claude to analyze the page and find the target element
   - Implements step caching to avoid repeated AI calls

2. **Assertion Agent** (`assertion-agent.ts`):
   - Input: assertion description + page state (DOM + screenshot)
   - Output: pass/fail + explanation
   - For AI Check steps ("verify the cart shows 3 items")

3. **Failure Agent** (`failure-agent.ts`):
   - Input: failed step info + screenshot + error + page state
   - Output: root cause analysis markdown
   - Runs automatically when a step fails

4. Integrate agents into the execution engine step handlers

**Files**: `src/lib/ai/*`, updates to `src/lib/engine/step-handlers.ts`.

### Phase 5: Dashboard & Polish
**Goal**: Test run history, analytics, and polished UX.

1. Run history page with filters (status, date, test)
2. Test health indicators (pass rate, flakiness detection)
3. Screenshot comparison (golden file diffing)
4. Bulk test operations
5. Environment/variable management

## Key Technical Patterns

### Real-Time Execution Updates (SSE)
```
POST /api/runs          → Creates run, starts execution in background
GET  /api/runs/:id/stream → SSE stream of step results
```
The executor writes to DB and emits events. The SSE endpoint reads from a pub/sub channel (in-process EventEmitter for MVP, upgradeable to Redis pub/sub).

### AI Element Targeting Flow
```
1. User step: "Click the 'Add to Cart' button"
2. Check StepCache for cached selector
3. If cache hit → use cached selector
4. If cache miss:
   a. Get page accessibility tree via Playwright
   b. Take screenshot
   c. Send to Claude: "Find the element matching 'Add to Cart button'"
   d. Claude returns selector + confidence
   e. Cache the result in StepCache
5. Execute Playwright action with resolved selector
Step Configuration Schema
Each step type has typed config:

NAVIGATE: { url: string }
CLICK: { target: string } (natural language, resolved by AI)
TYPE: { target: string, value: string }
WAIT: { duration: number } or { target: string }
AI_CHECK: { assertion: string }
AI_EXTRACT: { query: string, schema?: object }
JAVASCRIPT: { code: string }
